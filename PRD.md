## 1. 개요

### 1.1 한 줄 정의
회의실(1~6번)의 예약 현황을 한눈에 볼 수 있는 대시보드에서, **비어 있는 시간대를 클릭하면 바로 예약할 수 있는** 사내 회의실 예약 웹사이트.

### 1.2 배경 / 문제 정의
- 현재는 엑셀 시트(`회의실명단`, `예약현황`)로 회의실 정보와 예약을 수기 관리하고 있음.
- 엑셀 방식의 한계:
  - 실시간으로 누가 예약했는지 확인하기 어려움 (파일 공유/버전 충돌).
  - 특정 시간대가 비어 있는지 한눈에 파악하기 어려움.
  - 예약 생성/취소 시 셀을 직접 입력해야 해서 실수(중복 예약 등)가 발생하기 쉬움.
- coroom은 이 엑셀 데이터를 대체하는 웹 서비스로, **현황 조회(가시성)** 와 **예약 생성(클릭 한 번)** 을 핵심 가치로 삼는다.

### 1.3 목표 (MVP 범위)
1. 회의실 6개(1~6번)의 하루 단위 예약 현황을 그리드(시간 × 회의실)로 시각화한다.
2. 빈 시간 셀을 클릭하면 예약 폼이 뜨고, 제출 즉시 그리드에 반영된다.
3. 예약은 Supabase(Postgres)에 저장되고, 다른 사용자 화면에도 실시간으로 반영된다.
4. 예약 확정/취소 상태를 관리할 수 있다.

### 1.4 비목표 (Out of Scope, MVP 제외)
- 로그인/사내 계정 연동 (SSO) — MVP는 이름 직접 입력 방식으로 시작, 향후 확장 항목으로 분리.
- 반복 예약(매주 반복 등), 회의실 대여 승인 워크플로우.
- 이메일/슬랙 알림.
- 회의실 자체의 예약(위치, 사진) 외 부가 편의시설 예약(빔프로젝터 대여 등 별도 자원 예약).

## 2. 사용자 및 시나리오

### 2.1 사용자
- **임직원(예약자)**: 회의실을 조회하고 예약/취소하는 주 사용자. 별도 로그인 없이 이름/부서를 입력.
- **관리자(선택, Phase 2)**: 회의실 정보(수용 인원, 장비, 비고) 관리, 전체 예약 현황 관리.

### 2.2 핵심 시나리오
1. **현황 확인**: 사용자가 사이트에 접속하면 오늘 날짜 기준으로 6개 회의실 × 시간대 그리드가 보인다. 예약된 칸은 회의 제목/예약자가 표시되고, 빈 칸은 비어있음이 명확히 구분된다.
2. **빠른 예약**: 사용자가 원하는 회의실의 빈 시간 칸을 클릭 → 예약 폼(모달)이 뜸 → 이름/부서/회의제목/시작-종료시간 입력 후 등록 → 그리드에 즉시 반영.
3. **날짜 이동**: 사용자가 날짜를 다음/이전으로 이동하며 다른 날짜의 예약 현황을 확인.
4. **예약 취소**: 본인이 등록한 예약 칸을 클릭하면 상세 정보가 보이고, 취소 버튼으로 상태를 "취소"로 변경(그리드에서 다시 빈 칸으로 표시).
5. **회의실 정보 확인**: 회의실명 클릭/hover 시 수용 인원, 층, 보유 장비, 비고(예: "임원 보고용 우선 배정", "면접/촬영 겸용")를 확인.

### 3.2 예약 데이터 필드 — `예약현황` 시트 기준

| 컬럼 | 설명 | 예시 |
|---|---|---|
| 예약번호 | 예약 고유 ID | B2026071 |
| 회의실번호 | rooms.room_no FK | 3 |
| 예약자 | 예약한 사람 이름 | 박도윤 |
| 부서 | 예약자 소속 부서 | 마케팅팀 |
| 회의제목 | 회의 목적/제목 | 주간 팀 회의 |
| 예약일자 | 날짜 | 2026-07-13 |
| 시작시간 / 종료시간 | 예약 시간대 | 16:00 / 18:00 |
| 상태 | 확정 / 취소 | 확정 |

- 상태값은 **확정(confirmed) / 취소(cancelled)** 2종만 존재 (엑셀 예시 26건 모두 이 두 값). MVP는 이 2단계 상태만 지원한다.
- 부서 예시: 마케팅팀, 영업팀, 인사팀, 총무팀, 경영지원팀 → 초기에는 자유 텍스트 입력으로 처리하고, 이후 드롭다운(사전 정의 목록)으로 전환 가능.

## 4. 핵심 기능 명세

### 4.1 회의실 예약 현황 대시보드 (메인 화면)
- **레이아웃**: 세로축 = 시간(예: 09:00~18:00, 30분 또는 1시간 단위), 가로축 = 회의실 1~6번. 시간×회의실 그리드.
- **날짜 선택**: 상단에 날짜 네비게이션(◀ 오늘 ▶) 및 날짜 피커.
- **셀 상태 표현**:
  - 빈 셀: 클릭 가능, hover 시 "예약하기" 표시.
  - 예약된 셀(확정): 회의 제목 + 예약자 표시, 색상으로 구분. 여러 시간대에 걸친 예약은 셀 병합(rowspan)으로 표시.
  - 취소된 예약: 그리드에는 표시하지 않음(빈 칸으로 취급) — 단, 예약 이력 조회 시에는 확인 가능.
  - 과거 시간대(현재 시각 이전): 예약 불가 표시(회색 처리).
- **회의실 헤더**: 회의실명 + 수용인원 아이콘. 클릭/hover 시 층, 보유 장비, 비고 툴팁 표시.

### 4.2 예약 생성 (빈 시간 클릭)
- 그리드의 빈 셀 클릭 → 예약 모달 오픈.
- 모달 입력 필드: 회의실(자동 선택됨, 변경 가능), 예약일자(자동 선택됨), 시작시간, 종료시간, 예약자 이름, 부서, 회의 제목.
- **검증 규칙**:
  - 시작시간 < 종료시간.
  - 같은 회의실·같은 시간대에 이미 확정된 예약과 겹치면 등록 불가(중복 예약 방지) — 서버(DB) 레벨에서도 제약조건으로 이중 방어.
  - 필수 입력: 예약자, 부서, 회의 제목, 시작/종료시간.
- 등록 성공 시 모달 닫히고 그리드 즉시 갱신(Supabase Realtime 구독으로 다른 사용자 화면도 갱신).

### 4.3 예약 상세 조회 / 취소
- 예약된 셀 클릭 → 상세 정보 패널(예약자, 부서, 제목, 시간, 상태).
- "예약 취소" 버튼 → 상태를 취소로 변경(soft delete, row는 유지 — 이력 보존).
- (Phase 2) 본인 예약만 취소 가능하도록 제한 — MVP는 로그인이 없으므로 우선 전체 허용, 추후 인증 도입 시 강화.

### 4.4 회의실 정보 관리 (Phase 2, 관리자)
- 회의실 목록/수용인원/장비/비고 CRUD 화면.
- MVP에서는 시드 데이터로 고정(위 3.1 표), 화면상 조회만 가능.

### 4.5 필터/검색 (Phase 2)
- 수용 인원 기준 필터("4인 이상 회의실만 보기").
- 예약자/부서/회의 제목 검색.
- 내 예약만 모아보기.

## 6. 데이터 모델 (Supabase / PostgreSQL)

### 6.1 `rooms` 테이블

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | int (PK) | 회의실번호 (1~6) |
| name | text | 회의실명 (예: "1번 회의실 (소회의실 A)") |
| capacity | int | 수용인원 |
| floor | text | 층 (예: "3층") |
| equipment | text | 보유 장비 (콤마 구분 문자열 또는 text[]) |
| note | text (nullable) | 비고 |
| created_at | timestamptz | 생성일시 |

### 6.2 `reservations` 테이블

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid (PK, default gen_random_uuid()) | 예약 고유 ID |
| reservation_no | text (unique) | 표시용 예약번호 (예: B2026071, 앱에서 생성) |
| room_id | int (FK → rooms.id) | 예약 회의실 |
| reserver_name | text | 예약자 |
| department | text | 부서 |
| title | text | 회의 제목 |
| reservation_date | date | 예약일자 |
| start_time | time | 시작시간 |
| end_time | time | 종료시간 |
| status | text (check: 'confirmed' \| 'cancelled') | 상태 (확정/취소) |
| created_at | timestamptz | 생성일시 |
| updated_at | timestamptz | 수정일시 |

- **중복 예약 방지**: `EXCLUDE USING gist` 제약(+ `btree_gist` extension)으로 같은 `room_id`, 같은 `reservation_date`에서 `status = 'confirmed'`인 예약끼리 `[start_time, end_time)` 구간이 겹치지 않도록 DB 레벨에서 강제.
- **Realtime**: `reservations` 테이블에 Supabase Realtime(Postgres Changes) 구독을 켜서, 한 사용자가 예약/취소하면 다른 접속자 화면도 자동 갱신.
- **RLS**: MVP는 로그인이 없으므로 `rooms`는 전체 공개 read, `reservations`는 anon 권한으로 read/insert/update 허용(단, update는 상태를 'cancelled'로 바꾸는 용도로만 제한하는 정책 고려). Phase 2에서 인증 도입 시 "본인 예약만 취소 가능" 정책으로 강화.

## 7. 비기능 요구사항
- **실시간성**: Supabase Realtime으로 그리드가 폴링 없이 자동 갱신.
- **반응형**: 데스크톱 우선, 태블릿에서도 그리드 가로 스크롤로 대응.
- **성능**: 하루치 예약 조회는 회의실 6개 × 하루이므로 데이터량이 적어 별도 캐싱 불필요.
- **접근성**: 예약 가능/불가능 상태를 색상 외에 텍스트/아이콘으로도 구분.

## 8. 기술 스택 (제안)
- **프론트엔드**: HTML/CSS/JavaScript + `supabase-js` 클라이언트 (SPA, 별도 백엔드 서버 없이 Supabase 직접 호출).
- **데이터베이스/백엔드**: Supabase (PostgreSQL + Realtime + Auth(향후) + Row Level Security).
- **배포**: 정적 호스팅(Vercel / Netlify 등) — 백엔드 로직이 없으므로 정적 파일 배포로 충분.

## 10. 미해결 이슈 (Open Questions)
1. 예약 가능 시간대 범위(예: 09:00~18:00 고정 vs 24시간)와 최소 예약 단위(30분 vs 1시간)를 어떻게 정할지.
2. 인증 없이 이름을 자유 텍스트로 입력하게 할 경우, 타인의 예약을 임의로 취소하는 것을 어떻게 막을지 (Phase 1에서는 비밀번호/PIN 간단 확인 등 보완책이 필요할 수 있음).
3. `예약번호`(B2026071 형식)를 어떤 규칙으로 자동 채번할지 (연도+순번 등).
4. 5번 회의실의 "임원 보고용 우선 배정" 비고를 시스템 로직으로 반영할지(예: 예약 시 경고 문구만 표시) 여부.
